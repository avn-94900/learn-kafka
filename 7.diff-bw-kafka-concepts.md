## 1. Topic vs Partition

| Aspect | Topic | Partition |
|---|---|---|
| What it is | Logical category of messages | Physical subdivision of a topic |
| Purpose | Organise data by type | Enable parallel processing |
| Created by | User / Admin | Defined at topic creation time |
| Parallelism control | No | Yes |
| Ordering guarantee | No | Yes — within partition only |
| Storage location | Spread across partitions | Stored as an ordered, append-only log |
| Count | One per logical stream | Configurable per topic |

### Common Mistake

Assuming ordering exists across the entire topic.

```
❌ Topic ordering: Message1 → Message2 → Message3 (NOT guaranteed)
✅ Partition ordering: within partition-0, offset 0 → 1 → 2 (guaranteed)
```

> Ordering is **per partition**, never per topic.

## 2. Consumer Group vs Partition

| Aspect | Consumer Group | Partition |
|---|---|---|
| Type | Logical grouping of consumers | Physical data unit |
| Purpose | Load sharing across consumers | Parallel storage and processing |
| Relationship | Consumes from partitions | Assigned to one consumer per group |
| Key rule | One partition → one consumer per group | Multiple partitions allowed per topic |
| Scaling factor | Add consumers (up to partition count) | Add partitions |
| Idle scenario | Consumers > partitions → idle consumers | Partitions > consumers → one consumer handles multiple |

### Common Mistake

Adding more consumers than partitions expecting more parallelism.

```
Topic: orders (3 partitions)

✅ 3 consumers in group → each gets 1 partition
❌ 5 consumers in group → 2 consumers sit idle, no benefit
```

> Max useful consumers per group = number of partitions.

## 3. Offset vs Message Key

| Aspect | Offset | Message Key |
|---|---|---|
| What it is | Sequential position of a message in a partition | Routing identifier attached by producer |
| Generated by | Kafka broker (auto-assigned) | Producer (application code) |
| Purpose | Track read progress | Determine which partition to write to |
| Used by | Consumer | Producer |
| Ordering impact | Maintains read sequence | Maintains write sequence per key |
| Stored in | `__consumer_offsets` topic | Message header |
| Mutable | No | No |

### Common Mistake

Thinking offset controls which partition a message goes to.

```
❌ Offset → decides partition routing
✅ Key    → decides partition routing (hash(key) % num.partitions)
✅ Offset → tracks read position within a partition
```

## 4. At-Least-Once vs At-Most-Once vs Exactly-Once

| Delivery Mode | Data Loss | Duplicates | Performance | Config Required | Typical Use |
|---|---|---|---|---|---|
| At-Most-Once | Possible | No | Fastest | `acks=0`, auto-commit | Logs, metrics |
| At-Least-Once | No | Possible | Medium | `acks=all`, manual commit | Most business apps |
| Exactly-Once | No | No | Slowest | `enable.idempotence=true` + transactions | Financial transactions |

### Common Mistake

Assuming Kafka is exactly-once by default.

```
❌ Default Kafka → Exactly-Once
✅ Default Kafka → At-Least-Once
```

### Exactly-Once Configuration

```java
// Producer
props.put("enable.idempotence", true);
props.put("transactional.id", "order-producer-1");

producer.initTransactions();
producer.beginTransaction();
producer.send(record);
producer.commitTransaction();
```

```properties
# Consumer
isolation.level=read_committed
```

> Exactly-once requires both idempotent producer AND `isolation.level=read_committed` on consumer.

## 5. Producer Retry vs Idempotent Producer

| Aspect | Producer Retry | Idempotent Producer |
|---|---|---|
| Purpose | Resend failed messages | Prevent duplicate writes on retry |
| Problem it solves | Message loss on transient failure | Duplicate records from retries |
| Problem it creates | Can create duplicates | None |
| Required for ordering | Yes | Yes |
| Key config | `retries=Integer.MAX_VALUE` | `enable.idempotence=true` |
| Works together | Yes — must be used together | Yes — must be used together |

### Common Mistake

Enabling retries without idempotence.

```
❌ retries=MAX + enable.idempotence=false → duplicates on retry
✅ retries=MAX + enable.idempotence=true  → safe retry, no duplicates
```

```properties
# Correct combination
retries=2147483647
enable.idempotence=true
max.in.flight.requests.per.connection=1
```

> `enable.idempotence=true` automatically sets `acks=all` and `max.in.flight.requests.per.connection=5` (safe with idempotence). Set it to `1` only when strict ordering is needed.

## 6. Auto Commit vs Manual Commit

| Aspect | Auto Commit | Manual Commit |
|---|---|---|
| Offset control | Kafka handles automatically | Application controls explicitly |
| Commit timing | Every `auto.commit.interval.ms` (default 5s) | After successful processing |
| Risk | Message loss — commits before processing | Duplicate risk if commit is skipped |
| Reliability | Low | High |
| Ordering safety | Weak | Strong |
| Config | `enable.auto.commit=true` | `enable.auto.commit=false` |

### Common Mistake

Using auto-commit in business-critical systems.

```java
// ❌ Auto-commit: offset committed before processing finishes
props.put("enable.auto.commit", true);

// ✅ Manual commit: offset committed after processing
props.put("enable.auto.commit", false);

for (ConsumerRecord<String, String> record : records) {
    processOrder(record.value()); // process first
}
consumer.commitSync(); // then commit
```

### commitSync vs commitAsync (within Manual Commit)

| | `commitSync()` | `commitAsync()` |
|---|---|---|
| Blocks thread | Yes | No |
| Retries on failure | Yes | No |
| Use when | Ordering critical | High throughput |

## 7. Rebalance vs Restart

| Aspect | Rebalance | Restart |
|---|---|---|
| Trigger | Consumer joins or leaves the group | Consumer crashes or is restarted |
| Purpose | Redistribute partition assignments | Resume consumption from last offset |
| Impact | Temporary processing pause for all consumers | Only the restarting consumer is affected |
| Risk | Processing delay, duplicate reads | Duplicate reads if offset not committed |
| Controlled by | Group coordinator (broker) | Application / deployment |
| Frequency | On every group membership change | On crash or redeploy |

### Common Mistake

Ignoring rebalance impact — all consumers in the group pause during rebalance, not just the one that joined/left.

```
Group: [C1, C2, C3] consuming [P0, P1, P2]

C4 joins →
  Rebalance triggered →
    ALL consumers stop polling →
      Partitions reassigned →
        ALL consumers resume
```

> Use `CooperativeStickyAssignor` to minimise rebalance disruption (see section 20).

## 8. Key-Based Partitioning vs Round-Robin Partitioning

| Aspect | Key-Based | Round-Robin |
|---|---|---|
| Routing method | `hash(key) % num.partitions` | Cyclic distribution across partitions |
| Ordering | Preserved per key | Not preserved |
| Load balance | Depends on key distribution (can be skewed) | Even distribution |
| Use case | Business transactions, ordered events | Log collection, metrics |
| Key required | Yes | No (null key) |

### Common Mistake

Using round-robin (null key) when ordering is required.

```java
// ❌ Round-robin: no ordering guarantee
producer.send(new ProducerRecord<>("orders", null, orderData));

// ✅ Key-based: same orderId always goes to same partition
producer.send(new ProducerRecord<>("orders", orderId, orderData));
```

### Hot Partition Problem

```
❌ Bad key: all messages use same key → all go to one partition → hot partition
✅ Good key: high-cardinality key (orderId, userId) → even distribution
```

## 9. Consumer Lag vs Throughput

| Aspect | Consumer Lag | Throughput |
|---|---|---|
| Meaning | Number of messages not yet consumed | Messages processed per unit of time |
| Indicator | System delay / backlog | System processing capacity |
| Measured by | `latest offset - committed offset` | Records per second |
| Problem signal | Increasing lag | Low throughput |
| Root cause | Slow consumer or too few consumers | Under-provisioned consumers |
| Fix | Scale consumers or optimise processing | Add partitions + consumers |

### Common Mistake

Treating increasing lag as only a performance issue — it can also be a partition scaling issue.

```
Lag increasing + throughput normal → need more partitions + consumers
Lag increasing + throughput low   → consumer processing is too slow
```

### Monitoring

```bash
# Check consumer lag
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --describe \
  --group order-group
```

## 10. Retention vs Log Compaction

| Aspect | Retention | Log Compaction |
|---|---|---|
| Purpose | Delete old messages after time/size limit | Keep only the latest value per key |
| Trigger | Time (`retention.ms`) or size (`retention.bytes`) | Key updates |
| Data removed | All messages older than threshold | Older versions of same key |
| Use case | Event streams, audit logs | State storage, changelog topics |
| Message ordering | Preserved | Preserved |
| Null value behaviour | N/A | Null value = tombstone (deletes key) |

### Common Mistake

Using retention when compaction is needed (or vice versa).

```
✅ User profile updates  → log.cleanup.policy=compact
   (keep latest profile per userId)

✅ Transaction logs      → log.cleanup.policy=delete
   (keep last 7 days of transactions)

✅ Both needed           → log.cleanup.policy=compact,delete
```

```bash
# Set compaction on topic
kafka-topics.sh --alter \
  --topic user-profiles \
  --config log.cleanup.policy=compact \
  --bootstrap-server localhost:9092
```

## 11. Leader vs Follower

| Aspect | Leader | Follower |
|---|---|---|
| Handles writes | Yes — all writes go to leader | No |
| Handles reads | Yes (default) | Yes (with `replica.selector.class` config) |
| Failure role | New leader elected from ISR | Becomes leader candidate |
| Responsibility | Serve producer and consumer requests | Replicate data from leader |
| Count per partition | Exactly 1 | `replication.factor - 1` |

### Common Mistake

Assuming followers handle writes or that reads are load-balanced across replicas by default.

```
❌ Producer → writes to any broker
✅ Producer → writes only to partition leader

❌ Consumers read from followers by default
✅ Consumers read from leader by default
   (follower reads require explicit config: replica.selector.class)
```

## 12. poll() vs fetch()

| Aspect | `poll()` | `fetch()` |
|---|---|---|
| Triggered by | Consumer application code | Kafka client internals |
| Role | Request records from broker and return to app | Transfer data from broker to client buffer |
| Visibility | Application level | Broker / network level |
| Blocking | Yes — blocks until records available or timeout | No — background operation |
| Config | `max.poll.records`, `poll.timeout.ms` | `fetch.min.bytes`, `fetch.max.wait.ms` |

### Common Mistake

Thinking `poll()` reads directly from disk on every call.

```
Reality:
  Kafka client pre-fetches data into a local buffer (fetch)
  poll() returns records from that buffer
  If buffer is empty, poll() triggers a new fetch request
```

### poll() Timeout Danger

```java
// ❌ Long processing inside poll loop → triggers rebalance
while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        Thread.sleep(10000); // too slow → max.poll.interval.ms exceeded → rebalance
    }
}

// ✅ Keep poll loop fast, offload heavy processing
while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        executor.submit(() -> processRecord(record)); // async processing
    }
    consumer.commitSync();
}
```

## 13. commitSync() vs commitAsync()

| Aspect | `commitSync()` | `commitAsync()` |
|---|---|---|
| Blocks thread | Yes — waits for broker acknowledgment | No — fire and forget |
| Retries on failure | Yes — automatic | No — must handle manually |
| Performance | Lower | Higher |
| Risk | Slower throughput | Offset may not be committed on failure |
| Use when | Ordering critical, low throughput acceptable | High throughput, duplicates tolerable |

### Common Mistake

Using `commitAsync()` alone without a fallback `commitSync()` on shutdown.

```java
// ✅ Best practice: async during processing, sync on close
try {
    while (running) {
        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
        for (ConsumerRecord<String, String> record : records) {
            processRecord(record.value());
        }
        consumer.commitAsync(); // fast, non-blocking
    }
} finally {
    consumer.commitSync(); // ensure final offset is committed on shutdown
    consumer.close();
}
```

## 14. ISR vs OSR

| Aspect | ISR (In-Sync Replicas) | OSR (Out-of-Sync Replicas) |
|---|---|---|
| Definition | Replicas fully caught up with leader | Replicas lagging behind leader |
| Eligible for leader election | Yes | No |
| Write acknowledgment | Counted for `acks=all` | Not counted |
| Controlled by | `replica.lag.time.max.ms` | Falls out when lag exceeds threshold |
| Risk | Shrinking ISR reduces durability | OSR replica elected = data loss |

### Common Mistake

Setting `min.insync.replicas` too low or equal to 1.

```
replication.factor=3, min.insync.replicas=1
→ Leader alone can acknowledge writes
→ If leader fails before replication → data loss

✅ replication.factor=3, min.insync.replicas=2
→ At least 2 replicas must confirm → safe
```

```
ISR shrinks below min.insync.replicas
→ Producer gets NotEnoughReplicasException
→ Prevents data loss by refusing writes
```

## 15. Replication Factor vs min.insync.replicas

| Aspect | `replication.factor` | `min.insync.replicas` |
|---|---|---|
| What it controls | Total number of copies of each partition | Minimum replicas that must acknowledge a write |
| Set at | Topic creation | Topic config or broker config |
| Affects | Durability and availability | Write safety |
| Default | 1 (single broker) | 1 |
| Recommended | 3 | 2 |

### Common Mistake

Confusing replication factor with write safety.

```
replication.factor=3 → 3 copies exist
min.insync.replicas=1 → only 1 needs to confirm write → still risky

✅ replication.factor=3 + min.insync.replicas=2
   → 3 copies + at least 2 must confirm → safe and durable
```

### Formula

```
min.insync.replicas should be: replication.factor / 2 + 1

For RF=3 → min.insync.replicas=2
For RF=5 → min.insync.replicas=3
```

## 16. Consumer Rebalance vs Partition Reassignment

| Aspect | Consumer Rebalance | Partition Reassignment |
|---|---|---|
| What changes | Which consumer handles which partition | Which broker holds which partition |
| Triggered by | Consumer joins, leaves, or crashes | Broker added, removed, or manual reassignment |
| Affects | Consumer group | Kafka cluster topology |
| Downtime | Brief pause in consumption | Can be done with zero downtime |
| Controlled by | Group coordinator (broker) | Admin / `kafka-reassign-partitions.sh` |

### Common Mistake

Treating these as the same operation — they happen at completely different levels.

```
Consumer Rebalance:
  C1 crashes → Group coordinator reassigns P0 to C2

Partition Reassignment:
  Broker B1 removed → P0 moved from B1 to B3
  (consumers are unaffected if done correctly)
```

## 17. Dead Letter Queue vs Retry Topic

| Aspect | Dead Letter Queue (DLQ) | Retry Topic |
|---|---|---|
| Purpose | Store permanently failed messages | Temporarily hold messages for retry |
| Retry attempts | None — terminal destination | Configurable retry count |
| Processing | Manual inspection / alerting | Automatic re-processing |
| Use when | Message cannot be processed at all | Transient failure (DB down, timeout) |
| Example topic name | `orders.DLT` | `orders.retry-1`, `orders.retry-2` |

### Common Mistake

Sending all failed messages directly to DLQ without retry logic.

```
✅ Correct flow:
   orders → failure → orders.retry-1 (wait 1s)
                    → orders.retry-2 (wait 5s)
                    → orders.retry-3 (wait 30s)
                    → orders.DLT    (give up)

❌ Wrong flow:
   orders → any failure → orders.DLT (no retry attempted)
```

### Spring Kafka Example

```java
@Bean
public DefaultErrorHandler errorHandler(KafkaTemplate<String, String> template) {
    DeadLetterPublishingRecoverer recoverer = 
        new DeadLetterPublishingRecoverer(template);
    
    ExponentialBackOffWithMaxRetries backOff = 
        new ExponentialBackOffWithMaxRetries(3);
    backOff.setInitialInterval(1000L);
    backOff.setMultiplier(2.0);
    
    return new DefaultErrorHandler(recoverer, backOff);
}
```

## 18. Kafka Streams vs Kafka Consumer

| Aspect | Kafka Streams | Kafka Consumer |
|---|---|---|
| Type | Stream processing library | Low-level consumer API |
| Input | Kafka topic | Kafka topic |
| Output | Kafka topic (or state store) | Application logic (DB, API, etc.) |
| State management | Built-in (RocksDB) | Manual |
| Joins / aggregations | Built-in operators | Manual implementation |
| Fault tolerance | Built-in (changelog topics) | Manual |
| Use when | Stream processing, transformations, joins | Simple consumption and forwarding |

### Common Mistake

Using raw `KafkaConsumer` for complex stream processing that Kafka Streams handles natively.

```java
// ❌ Manual join with KafkaConsumer — complex and error-prone
// ✅ Kafka Streams join — built-in

KStream<String, Order> orders = builder.stream("orders");
KTable<String, Customer> customers = builder.table("customers");

KStream<String, EnrichedOrder> enriched = orders.join(
    customers,
    (order, customer) -> new EnrichedOrder(order, customer)
);
enriched.to("enriched-orders");
```

## 19. Schema Registry vs Manual Serialization

| Aspect | Schema Registry (Avro/Protobuf) | Manual Serialization (JSON/String) |
|---|---|---|
| Schema enforcement | Yes — validated on produce/consume | No |
| Schema evolution | Supported (backward/forward compatible) | Manual, error-prone |
| Payload size | Smaller (binary format) | Larger (text format) |
| Consumer safety | Schema mismatch caught at registry | Runtime deserialization error |
| Setup complexity | Higher | Lower |
| Use when | Multiple teams, evolving schemas | Simple internal use |

### Common Mistake

Using JSON without schema validation across multiple services — schema changes break consumers silently.

```
❌ Producer adds new required field to JSON
   → Consumer deserializes without that field
   → NullPointerException at runtime

✅ Schema Registry with backward compatibility
   → New field must have default value
   → Old consumers continue working
```

## 20. Eager Rebalance vs Cooperative Rebalance

| Aspect | Eager Rebalance (default) | Cooperative (Incremental) Rebalance |
|---|---|---|
| Strategy | Stop-the-world — all partitions revoked | Only affected partitions reassigned |
| Processing pause | All consumers pause | Only consumers with reassigned partitions pause |
| Performance impact | High — full group stops | Low — minimal disruption |
| Config | `partition.assignment.strategy=RangeAssignor` | `partition.assignment.strategy=CooperativeStickyAssignor` |
| Available since | Always | Kafka 2.4+ |

### Common Mistake

Using default eager rebalance in high-throughput systems — causes unnecessary full pauses.

```properties
# ✅ Use cooperative rebalance for production
partition.assignment.strategy=org.apache.kafka.clients.consumer.CooperativeStickyAssignor
```

```
Eager:
  C1, C2, C3 all stop → partitions reassigned → all resume
  (entire group pauses even if only C4 joined)

Cooperative:
  Only partitions moving to C4 are revoked
  C1, C2, C3 continue processing unaffected partitions
```
