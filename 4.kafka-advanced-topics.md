# Kafka Advanced Topics



## 1. Custom Partitioning

By default, Kafka assigns messages to partitions using the message key hash. You can override this behavior using three methods.

### Three Partitioning Methods

| Method | How It Works | When to Use |
| --- | --- | --- |
| Direct Partition Assignment | You specify the partition number explicitly | When you need full control |
| Key-based Hashing | Kafka hashes the key to pick a partition | Default behavior, consistent routing |
| Custom Partitioner | Your own class decides the partition | Business-specific routing rules |

---

### Method 1 - Direct Partition Assignment

You pass the partition number directly when creating the record.

```java
// Send to partition 0 explicitly
ProducerRecord<String, String> record1 = new ProducerRecord<>(
    topic,      // topic name
    0,          // partition number
    "key1",     // message key
    "message1"  // message value
);

// Send to partition 2 explicitly
ProducerRecord<String, String> record2 = new ProducerRecord<>(
    topic,
    2,
    "key2",
    "message2"
);
```

---

### Method 2 - Key-Based Hashing (Default)

Kafka hashes the key and picks a partition automatically. Messages with the same key always go to the same partition.

```java
// Partition is determined by hash of the key
ProducerRecord<String, String> record1 = new ProducerRecord<>(
    topic,
    "user123",      // key - Kafka hashes this to pick a partition
    "user data 1"
);

ProducerRecord<String, String> record2 = new ProducerRecord<>(
    topic,
    "user456",      // different key - may go to a different partition
    "user data 2"
);
```

**Note:** Same key always lands on the same partition. This guarantees ordering per key.

---

### Method 3 - Custom Partitioner

You write a class that implements `Partitioner` and contains your own routing logic.

#### Step 1 - Register the custom partitioner in producer properties

```java
props.put("partitioner.class", "com.example.CustomPartitioner");
```

#### Step 2 - Implement the Partitioner interface

```java
class CustomPartitioner implements Partitioner {

    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
                         Object value, byte[] valueBytes, Cluster cluster) {

        int numPartitions = cluster.partitionCountForTopic(topic);

        if (key == null) {
            // Round-robin for null keys
            return (int) (System.currentTimeMillis() % numPartitions);
        }

        String keyStr = (String) key;

        // VIP users always go to partition 0
        if (keyStr.startsWith("vip-")) {
            return 0;
        }

        // Priority users go to partition 1 if available
        if (keyStr.startsWith("priority-") && numPartitions > 1) {
            return 1;
        }

        // Regular users use hash-based partitioning on remaining partitions
        int regularPartitionStart = Math.min(2, numPartitions - 1);
        int availablePartitions = numPartitions - regularPartitionStart;

        if (availablePartitions <= 0) {
            return 0;
        }

        return regularPartitionStart +
               (Utils.toPositive(Utils.murmur2(keyBytes)) % availablePartitions);
    }

    @Override
    public void close() { }

    @Override
    public void configure(Map<String, ?> configs) { }
}
```

#### Custom Partitioner Routing Logic Summary

| Key Prefix | Assigned Partition | Reason |
| --- | --- | --- |
| `vip-` | Partition 0 | High priority lane |
| `priority-` | Partition 1 | Medium priority lane |
| All others | Partition 2 and beyond | Distributed by hash |
| null key | Any partition | Round-robin |

## 2. Load Balancing with Partitions and Consumer Groups

### Concept Overview

Kafka uses partitions and consumer groups together to distribute work across multiple service instances.

| Concept | Analogy |
| --- | --- |
| Topic | A book of orders |
| Partition | A chapter in the book |
| Consumer group | A class of students reading the book |
| Consumer instance | One student in the class |

Each student in the same class reads different chapters. Two different classes can read the same book independently.

---

### Real-World Scenario - E-Commerce Platform

**Producer:** Checkout Service sends `OrderCreated` events to topic `OrdersTopic`.

**Topic:** `OrdersTopic` with 3 partitions (P1, P2, P3).

**Consumers (Services):**

| Consumer Group | Service | Instances |
| --- | --- | --- |
| Group1 | Order Service | C1, C2 |
| Group2 | Inventory Service | D1, D2 |
| Group3 | Notification Service | E1, E2 |

---

### Partition Assignment Per Group

Each group gets its own independent assignment of partitions.

**Order Service (Group1):**

```
P1 -> C1
P2 -> C2
P3 -> C1
```

C1 handles P1 and P3. C2 handles P2.

**Inventory Service (Group2):**

```
P1 -> D1
P2 -> D2
P3 -> D1
```

D1 handles P1 and P3. D2 handles P2.

**Notification Service (Group3):**

```
P1 -> E1
P2 -> E2
P3 -> E1
```

E1 handles P1 and P3. E2 handles P2.

**Key point:** All three groups read the same topic independently. Each group sees every message.

---

### How the Full Flow Works

1. Checkout Service produces `OrderCreated` events to `OrdersTopic`.
2. Kafka distributes messages across P1, P2, and P3.
3. Kafka assigns partitions to consumers within each group.
4. Each service instance reads only its assigned partitions.
5. Order Service, Inventory Service, and Notification Service all process the same events without blocking each other.

---

### Benefits of This Design

| Benefit | Explanation |
| --- | --- |
| Horizontal scalability | Add more instances to a consumer group to handle more load |
| Load balancing | Kafka redistributes partitions evenly across instances |
| Fault tolerance | If one instance crashes, its partitions are reassigned to healthy instances |
| Independent processing | Multiple services consume the same topic without interfering with each other |

---

### Scaling Rule

**Max useful consumers per group = Number of partitions.**

If you have 3 partitions and add a 4th consumer instance to a group, the 4th instance sits idle because there are no partitions left to assign to it.
