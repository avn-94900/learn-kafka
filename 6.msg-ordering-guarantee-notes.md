## 1. Overview

Kafka guarantees message ordering **within a single partition only**. To maintain strict ordering from producer → broker → consumer, specific configurations must be applied at all three levels.

### Ordering Guarantee Levels

| Level | Scope | Configuration |
|---|---|---|
| Global ordering | All messages in topic | Single partition (`num.partitions=1`) |
| Per-key ordering | Messages with same key | Multiple partitions + key-based routing |
| No ordering | Any message | Default Kafka behavior |

> Most production systems use **per-key ordering** to balance ordering guarantees with throughput.

## 2. Producer Configuration

### Mandatory Settings

| Property | Value | Purpose |
|---|---|---|
| `acks` | `all` | Leader waits for all in-sync replicas before acknowledging |
| `enable.idempotence` | `true` | Prevents duplicate writes during retries |
| `max.in.flight.requests.per.connection` | `1` | Prevents retry-based reordering |
| `retries` | `Integer.MAX_VALUE` | Retries until success without message loss |
| `retry.backoff.ms` | `100` | Delay between retries (avoids tight loops) |
| `key.serializer` | `StringSerializer` | Required for key-based partitioning |
| `value.serializer` | `StringSerializer` | Required for message serialization |

### Partitioning Strategy

Always send messages with a **key** to ensure consistent partition routing.

```java
// Use business key for ordering boundary
ProducerRecord<String, String> record = 
    new ProducerRecord<>("orders", orderId, message);

producer.send(record);
```

- Same key → same partition → order preserved
- Use business keys: `orderId`, `customerId`, `accountId`

### Why These Settings Matter

| Setting | Without It | With It |
|---|---|---|
| `acks=all` | Message loss on leader failure | Replicas confirm before ack |
| `enable.idempotence=true` | Duplicate messages on retry | Exactly-once semantics |
| `max.in.flight.requests.per.connection=1` | Message2 may arrive before Message1 | Strict ordering maintained |

## 3. Broker Configuration

### Topic-Level Settings

| Property | Value | Purpose |
|---|---|---|
| `num.partitions` | `1` for global ordering, `N` for per-key ordering | Controls parallelism |
| `replication.factor` | `3` (recommended) | Ensures durability |
| `min.insync.replicas` | `2` (recommended) | Prevents leader-only writes |

### Topic Creation Example

```bash
# Global ordering (single partition)
kafka-topics.sh \
  --create \
  --topic orders \
  --partitions 1 \
  --replication-factor 3 \
  --config min.insync.replicas=2 \
  --bootstrap-server localhost:9092

# Per-key ordering (multiple partitions)
kafka-topics.sh \
  --create \
  --topic orders \
  --partitions 10 \
  --replication-factor 3 \
  --config min.insync.replicas=2 \
  --bootstrap-server localhost:9092
```

### Leader Behavior

- Only the **leader** writes messages to the partition
- **Followers** replicate in the same order
- Leader failover preserves committed offsets
- In-sync replicas (ISR) maintain ordering during replication

## 4. Consumer Configuration

### Mandatory Settings

| Property | Value | Purpose |
|---|---|---|
| `enable.auto.commit` | `false` | Prevents early commits before processing |
| `max.poll.records` | `1` | Enforces sequential processing |
| `isolation.level` | `read_committed` | Avoids reading uncommitted messages |
| `fetch.min.bytes` | `1` | Immediate fetch without waiting |
| `group.id` | `order-group` | Required consumer group identifier |
| `key.deserializer` | `StringDeserializer` | Must match producer serializer |
| `value.deserializer` | `StringDeserializer` | Must match producer serializer |

### Threading Model Rules

| Rule | Reason |
|---|---|
| One consumer thread per partition | Multiple threads break ordering |
| No multi-threaded record processing | Parallel processing reorders messages |
| Process records sequentially | Ensures order is maintained |

### Offset Commit Strategy

```java
// Correct: Commit after processing
while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    
    for (ConsumerRecord<String, String> record : records) {
        processRecord(record); // Process first
    }
    
    consumer.commitSync(); // Then commit
}
```

```java
// Wrong: Commit before processing
consumer.commitSync(); // DON'T DO THIS
processRecord(record);
```

### Acknowledgment Rule

```
✅ Process → Commit offset
❌ Commit → Process
```

## 5. Partitioning Strategy

### Key Routing Rules

- Use a **deterministic key** (same input → same partition)
- Key must represent the **ordering boundary**
- Kafka uses `hash(key) % num.partitions` to determine partition

### Example Keys

| Use Case | Key |
|---|---|
| Order processing | `orderId` |
| User activity | `userId` or `customerId` |
| Account transactions | `accountId` |
| Device telemetry | `deviceId` |

### Single Partition vs Multiple Partitions

| | Single Partition | Multiple Partitions |
|---|---|---|
| Ordering scope | Global (all messages) | Per key only |
| Throughput | Low (single consumer) | High (parallel consumers) |
| Scalability | Limited | Scales with partition count |
| Use when | Strict global ordering required | High throughput + per-key ordering |

```java
// Single partition: global ordering
Properties props = new Properties();
props.put("partitions", 1);

// Multiple partitions: per-key ordering
Properties props = new Properties();
props.put("partitions", 10);
producer.send(new ProducerRecord<>("orders", customerId, message));
```

## 6. Failure and Retry Handling

### Producer Retry Behavior

| Configuration | Behavior |
|---|---|
| `retries=Integer.MAX_VALUE` | Retries indefinitely until success |
| `enable.idempotence=true` | Prevents duplicates during retries |
| `max.in.flight.requests.per.connection=1` | Prevents Message2 arriving before Message1 |

### Consumer Restart Behavior

- Consumer resumes from **last committed offset**
- No reordering if commits are sequential
- Uncommitted messages are reprocessed (at-least-once delivery)

### Broker Failure Handling

| Event | Behavior |
|---|---|
| Leader failure | New leader elected from ISR |
| Follower failure | Removed from ISR, rejoins after catching up |
| Network partition | Producer retries until leader is reachable |

- Leader election maintains committed order
- Followers replicate in the same sequence
- Committed offsets are never lost

## 7. Operational Constraints

### Must Never Be Violated

| Constraint | Reason |
|---|---|
| Never send messages without keys when ordering matters | Random partition assignment breaks ordering |
| Never use multiple threads per partition | Parallel processing reorders messages |
| Never enable auto-commit for strict ordering | May commit before processing completes |
| Never increase `max.in.flight.requests.per.connection` above 1 | Allows out-of-order delivery on retry |
| Never commit offsets before processing finishes | Message loss on consumer restart |
| Never change partition count after production begins | Breaks key-to-partition mapping |
| Never rebalance without committing offsets first | May cause duplicate processing |

```java
props.put("group.id", "order-group");
props.put("enable.auto.commit", false);
props.put("max.poll.records", 1);

props.put("value.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");

consumer.subscribe(Collections.singletonList("orders"));

while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    
    for (ConsumerRecord<String, String> record : records) {
        // Process record
        processOrder(record.value());
    }
    
    // Commit only after successful processing
    consumer.commitSync();
}
```

## 9. Trade-offs

### Strict Ordering Costs

| Aspect | Impact |
|---|---|
| Throughput | Lower (sequential processing) |
| Latency | Higher (no parallelism) |
| Scalability | Limited (single consumer per partition) |
| Complexity | Higher (careful configuration required) |

### Real-World Approach

Most production systems implement **per-key ordering** instead of global ordering:

```
✅ customerId → ordering preserved within customer
✅ All customers → parallel processing allowed
✅ High throughput + ordering guarantees
```

### Comparison

| Approach | Throughput | Ordering | Scalability |
|---|---|---|---|
| Global ordering (1 partition) | Low | Strict | Poor |
| Per-key ordering (N partitions) | High | Per key | Good |
| No ordering (N partitions, no keys) | Highest | None | Best |

### Interview Critical Point

> Strict ordering usually means lower throughput, higher latency, and less scalability. Real systems balance ordering requirements with performance by using per-key ordering across multiple partitions.
