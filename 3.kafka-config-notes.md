## Kafka Configuration


## 1. Message Retention Policy

Kafka can automatically delete old messages based on time or size. You configure this in `server.properties`.

### Key Properties

| Property | Description |
| --- | --- |
| `log.retention.ms` | How long to keep messages (in milliseconds) |
| `log.retention.bytes` | Maximum size of log data to keep |
| `log.cleanup.policy` | Cleanup strategy: `delete` or `compact` |

### How Kafka Applies Both Together

Kafka checks both conditions. Whichever limit is hit first triggers cleanup.

- If the message is older than `retention.ms`, Kafka deletes it.
- If the log size exceeds `retention.bytes`, Kafka deletes the oldest segments.

### Example Configuration

```properties
log.retention.ms=604800000       # 7 days
log.retention.bytes=1073741824   # 1 GB
log.cleanup.policy=delete
```

**Note:** These settings apply to all topics unless you override them per topic. You must restart the Kafka broker after changing `server.properties`.

## 2. Backpressure Handling

Backpressure happens when the Kafka broker is slow and the producer cannot send messages fast enough.

### Key Properties

| Property | Description |
| --- | --- |
| `buffer.memory` | How much memory the producer uses to buffer messages locally |
| `max.block.ms` | How long the producer waits if the buffer is full |

### Step-by-Step Behavior

1. Producer sends messages to Kafka.
2. If the broker is slow, messages are stored in the producer's memory buffer.
3. If the buffer fills up, the producer pauses and waits.
4. If the wait exceeds `max.block.ms`, the producer throws a `TimeoutException`.

### Example Configuration

```properties
buffer.memory=67108864   # 64 MB
max.block.ms=30000       # 30 seconds
```

### Example Exception

```java
org.apache.kafka.common.errors.TimeoutException:
Failed to allocate memory within the configured max blocking time 30000 ms
```

## 3. Consumer Lag

**Consumer lag** is the number of messages the consumer has not yet read.

### Formula

```
Consumer Lag = Latest Offset - Consumer's Current Offset
```

### Example

- Producer wrote offsets 100 to 200.
- Consumer has read up to offset 150.
- Lag = 200 - 150 = 50 messages.

### How to Monitor Lag

| Tool | Purpose |
| --- | --- |
| Prometheus + Grafana | Dashboards and alerting |
| Kafka Manager | Cluster management UI |
| Confluent Control Center | Enterprise monitoring |
| Kafka Exporter | Metrics export |

### Metric to Track

```
kafka.consumer.ConsumerFetcherManager.metrics.records-lag
```

### How to Manage Lag

**Tune consumer settings:**

| Property | Description |
| --- | --- |
| `max.poll.records` | How many messages to fetch per poll |
| `max.poll.interval.ms` | Max time allowed for processing before Kafka marks consumer as dead |
| `fetch.max.bytes` | Max bytes fetched per request |
| `session.timeout.ms` | Timeout before Kafka marks consumer as dead |

**Scale consumers:**

- Add more consumer instances to the consumer group.
- Each consumer handles one or more partitions, which reduces load per consumer.

**Optimize processing logic:**

- Use async processing or worker thread pools.
- Avoid long blocking calls inside the consumer loop.

**Increase retention:**

- If consumers are expected to lag, increase `log.retention.ms` so messages are not deleted before they are read.

## 4. Message Delivery Semantics

Kafka supports three delivery guarantees. Each has trade-offs.

### Overview Table

| Semantics | Duplicates | Message Loss | Use Case |
| --- | --- | --- | --- |
| At-most-once | No | Yes | Metrics, logging |
| At-least-once | Yes | No | General use |
| Exactly-once | No | No | Payments, orders |

---

### At-Most-Once

Messages may be lost. No retries. Fastest option.

**Producer config:**

```yaml
spring:
  kafka:
    producer:
      acks: 0
      retries: 0
      enable-idempotence: false
```

**Consumer config:**

```yaml
spring:
  kafka:
    consumer:
      enable-auto-commit: true
      auto-commit-interval: 1000
```

---

### At-Least-Once

Messages are never lost, but duplicates can occur if the producer retries.

**Producer config:**

```yaml
spring:
  kafka:
    producer:
      acks: all
      retries: 5
      enable-idempotence: false
```

**Consumer config:**

```yaml
spring:
  kafka:
    consumer:
      enable-auto-commit: false
      auto-offset-reset: earliest
```

You must commit offsets manually after processing:

```java
consumer.commitSync();
```

---

### Exactly-Once

No duplicates, no loss. Requires all three parts to work together.

```
Idempotent Producer + Transactions + Read Committed Consumer = Exactly-Once
```

**Producer config:**

```properties
enable.idempotence=true
acks=all
retries=2147483647
max.in.flight.requests.per.connection=5
transactional.id=txn-producer-1
```

**Consumer config:**

```properties
isolation.level=read_committed
enable.auto.commit=false
```

---

## 5. Core Parts of Exactly-Once Semantics (EOS)

### Part 1 - Idempotent Producer

Prevents duplicate messages when the producer retries.

How it works:
- Kafka assigns each producer a **Producer ID (PID)**.
- Each message gets a **sequence number**.
- Kafka rejects duplicate sequence numbers automatically.

```properties
enable.idempotence=true
acks=all
retries=2147483647
```

**Important:** Idempotent producer alone does not give you exactly-once. It only prevents duplicate writes.

---

### Part 2 - Transactions

Makes a group of messages act as one unit. Either all succeed or none do.

```
beginTransaction()
  send messages
commitTransaction()  OR  abortTransaction()
```

Kafka uses a **Transaction Coordinator** (a special internal broker component) to manage this.

```properties
enable.idempotence=true
transactional.id=txn-1
```

Without `transactional.id`, transactions do not work.

---

### Part 3 - Read Committed Consumer

The consumer reads only messages from committed transactions. It skips aborted ones.

```properties
isolation.level=read_committed
enable.auto.commit=false
```

---

## 6. End-to-End Exactly-Once Flow

### Success Path

1. Producer calls `beginTransaction()`.
2. Producer sends messages (not visible to consumers yet).
3. Producer calls `commitTransaction()`.
4. Kafka marks messages as committed.
5. Consumer reads only those committed messages.

### Failure Path

1. Producer calls `beginTransaction()`.
2. Producer sends messages.
3. An error occurs. Producer calls `abortTransaction()`.
4. Kafka marks messages as aborted.
5. Consumer never sees them.

## 7. Exactly-Once Code Examples

### Producer Example

```java
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("key.serializer", "org.apache.kafka.common.serialization.StringSerializer");
props.put("value.serializer", "org.apache.kafka.common.serialization.StringSerializer");
props.put("enable.idempotence", "true");
props.put("acks", "all");
props.put("transactional.id", "txn-producer-1");

KafkaProducer<String, String> producer = new KafkaProducer<>(props);
producer.initTransactions();

try {
    producer.beginTransaction();
    producer.send(new ProducerRecord<>("test-topic", "key1", "hello kafka"));
    producer.commitTransaction();
} catch (Exception e) {
    producer.abortTransaction();
} finally {
    producer.close();
}
```

### Consumer Example

```java
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("group.id", "txn-group");
props.put("key.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");
props.put("value.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");
props.put("isolation.level", "read_committed");
props.put("enable.auto.commit", "false");

KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
consumer.subscribe(Collections.singletonList("test-topic"));

while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        System.out.println(record.value());
    }
    consumer.commitSync();
}
```

```xml
</dependency>
```

## 8. Idempotency vs Deduplication

These two concepts are often confused. They are not the same.

| Feature | Idempotency | Deduplication |
| --- | --- | --- |
| Purpose | Prevent duplicate effects | Remove duplicate messages |
| Where applied | Producer or consumer logic | Consumer or storage layer |
| Logic | Same input produces same result | Detect and remove repeated messages |
| Needs storage | Usually no | Yes |
| Example | Placing same order twice creates one order | Remove duplicate order messages from stream |
| Kafka support | Built-in at producer level | Must be implemented by developer |
